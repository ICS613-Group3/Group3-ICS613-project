At the end of the July 9th class, the instructor provided detailed guidance on the manual testing assignment and how to construct **test cases** for your project.

### Focus and Scope of Manual Testing

* **Front-end Emphasis:** Manual testing should primarily focus on the **front-end**, specifically for features or interactions that cannot be easily automated 1\.  
* **System Readiness:** While testing the front-end, the **back-end should already be functional** to support those tests 2\.  
* **Prioritization:** You are not expected to write hundreds of cases; instead, focus on **core, critical test cases** that cover the most important features you have implemented or plan to implement 2\.

### How to Structure Test Cases

* **Based on User Stories:** Your test cases should be derived directly from the **acceptance criteria** you defined for your user stories 3\.  
* **Coverage:** You should include scenarios for the **"happy path"** (the successful standard flow), **negative paths** (error handling), and various **edge cases** 3\.  
* **Format:** The instructor recommends using a **spreadsheet** where each row represents a single test case 4\.

### Submission and Grading Details

* **Required Columns:** Each test case should include details from the **Test Case ID** through to the **Expected Result** 4, 5\.  
* **Fields to Leave Blank:** For the initial assignment submission, you must leave the **"Actual Result"** and **"Pass/Fail"** columns **blank** 4, 5\. These fields will be populated later by your team during the actual testing phase 4\.  
* **Definition of "Pass":** A test is marked as "Pass" only when the **expected result matches the actual result** recorded during testing 6\.
