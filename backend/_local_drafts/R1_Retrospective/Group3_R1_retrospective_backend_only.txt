Group 3

What went well (Backend):
- All 46 endpoints implemented across 7 routers: Auth (13), Tools (10), Reservations (10), Reviews (5), Notifications (2), Admin (5), Health (1)
- 187 unit tests — full coverage of happy paths, permissions, edge cases, audit logging, and notifications
- Authentication: JWT access/refresh token rotation, rate limiting on auth endpoints, email verification flow, invite-only registration
- Tool lifecycle: browse with category/search/availability filters, photo upload (1–5, magic-byte validated, max 5 MB), deactivate/reactivate with auto-cancel of pending reservations
- Reservation lifecycle: request -> approve/deny -> picked-up -> returned, double-booking prevention via GiST EXCLUDE constraint, damage reporting with 7-day window and borrower trust score impact
- Reviews: per-reservation (one per party), edit/delete within 24-hour window, GET /users/me/reviews with role=received|given filtering
- Notifications: auto-triggered on every state transition, unread-count badge support, bodies include tool name and party display names
- Admin: member list with search/filter/pagination, member suspend/reactivate/delete, audit log with action/actor filters and pagination, invite CRUD, admin force-return on reservations, admin tool deactivate/reactivate across all owners
- Background scheduler: auto-cancel un-picked-up reservations after grace period, overdue escalation, token cleanup
- Email integration: async email sending for invites, verification, password reset via SMTP (MailHog-compatible for dev)
- Post-R1 bug fixes integrated: damage attribution to borrower, duplicate damage guard, damage-rating integration (trust_score reduction), suspended login, 1-day rentals, reactivation reason field, account deletion with owner obligation checks, notification name fields, profile validation, response name fields, VerifyTokenError -> 400
- Docker compose dev environment with PostgreSQL + MailHog
- Seed data script for demo users

What did not go well:
- R1 bug fixes had to be done in 2 rounds post-demo (damage attribution fix + remaining R1 bugs), indicating gaps in edge-case testing before the initial submission
- Rate limit defaults (50/min) were left at generous development values and never tightened for production-readiness
- Some backend response shapes lacked fields
- Mock-to-backend field gaps (notesForBorrowers, availability, latestReturnTime) were discovered late and had to be dropped rather than properly implemented
- No shared enum mapping utility between backend enums and display strings, causing manual mapping on submit

What needs to be fixed before R2:
- Tighten rate limit defaults from 50/min to 5–10/min for production safety (backend config change, ~10 min)
- Resolve mock-field gaps (notesForBorrowers, availability, latestReturnTime) — decision: drop for R1/R2, handle as N/A
- Update API contract documentation to reflect the latest field names and response shapes after R1 bug fixes
- Implement admin reservation overview endpoint for R2 story US34
- Add tool-condition enum mapping helper to convert backend enums to display strings
- Document photo URL prefix convention (photos[n].url already includes /uploads/)
